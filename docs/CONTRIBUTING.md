# Contributing to Research Web Crawler

Thank you for your interest in contributing to Research Web Crawler!

## Code of Conduct

### Our Pledge

We are committed to providing a welcoming community for all.

### Our Standards

- Use welcoming and inclusive language
- Be respectful of differing viewpoints
- Accept constructive criticism gracefully
- Focus on what's best for the community
- Show empathy towards others

## Getting Started

### Prerequisites

Development environment for visionOS:
- Latest development tools
- Git and GitHub account
- Familiarity with visionOS development

### Fork and Clone

1. Fork the repository on GitHub
2. Clone your fork locally:

```bash
git clone https://github.com/YOUR_USERNAME/research-web-crawler.git
cd research-web-crawler
```

3. Add upstream remote:

```bash
git remote add upstream https://github.com/axion/research-web-crawler.git
```

## How to Contribute

### Reporting Bugs

Before creating bug reports, check existing issues.

When reporting:
- Clear title and description
- Steps to reproduce
- Expected vs actual behavior
- Screenshots if applicable
- Environment details (visionOS version, device, etc.)

### Suggesting Features

Feature requests welcome! Please:
- Check existing requests
- Provide clear use case
- Explain benefits
- Consider implementation

### Pull Requests

1. Create a feature branch:
```bash
git checkout -b feature/amazing-feature
```

2. Make your changes following coding standards

3. Commit your changes:
```bash
git commit -m "Add amazing feature"
```

4. Push to your fork:
```bash
git push origin feature/amazing-feature
```

5. Open a Pull Request

## Coding Standards

### General Guidelines

- Write clear, readable code
- Add comments for complex logic
- Use meaningful names
- Keep functions small and focused
- Follow DRY principle
- Handle errors appropriately

### Testing

- Write tests for new features
- Maintain >80% coverage
- Include unit and integration tests
- Test edge cases

## Review Process

### Timeline

- Initial review: 3-5 business days
- Follow-up: 2-3 business days

### What to Expect

1. Automated checks
2. Code review
3. Feedback and change requests
4. Approval and merge
5. Inclusion in next release
